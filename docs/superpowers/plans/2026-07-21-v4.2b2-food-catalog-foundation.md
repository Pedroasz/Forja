# V4.2B.2 — Plano de implementação da fundação do catálogo

Status: plano executável após as aprovações da V4.2B.1
Estratégia: TDD, migrações aditivas, commits pequenos, push após cada checkpoint validado
Fora de escopo: produção, dados reais, receitas, compartilhamento organizacional de alimentos privados, bulk OFF/TBCA e deploy.

## Pré-condições

1. Pedro aprova as cinco decisões da especificação.
2. A branch B.2 parte do `origin/main` vigente, não desta branch documental sem merge/rebase explícito.
3. A fonte piloto possui evidência de licença suficiente; caso contrário, testes usam fixtures sintéticas.
4. Docker/Supabase local estão disponíveis para validação de migration, RLS e pgTAP.
5. O problema preexistente de final de linha no teste de workspaces é corrigido em checkpoint separado ou classificado no PR com evidência reproduzível.

## Convenções de execução

- Cada checkpoint começa com teste falhando pelo comportamento ainda ausente.
- Não editar migrations já aplicadas; criar arquivos timestamped novos.
- Antes de push: lint/validador de migration, testes Node relevantes, pgTAP local e diff protegido.
- Antes de qualquer ação remota Supabase: novo checkpoint de aprovação; o plano padrão termina validado localmente.
- Tipos são regenerados após schema local e comparados com `supabase/database.types.ts`.
- Segredos entram apenas por variáveis locais ignoradas; nenhum artefato de fonte ou log sensível é commitado.

## Checkpoint B.2C — RLS, GRANTs e RPCs privadas

**Arquivos previstos**

- mesma migration B.2 ou migration de hardening imediatamente posterior
- `supabase/tests/food_catalog_access_test.sql`

**Matriz de testes**

1. `anon` não lê nem escreve nenhuma tabela.
2. autenticado pesquisa catálogo público por RPC, mas não faz escrita direta.
3. usuário A cria/lê/edita/desativa seu alimento privado por RPC; cada edição gera source record `user_entered`, perfil completo e revisão monotônica.
4. usuário B não descobre ID, alias, nutrientes ou medidas de A.
5. nutricionista sem ownership não obtém acesso adicional.
6. combinações inválidas de visibility/owner são rejeitadas e RPC não altera esses campos.
7. clientes não executam helpers/importadores nem selecionam tabelas de staging, payload ou batches.
8. papel PostgreSQL dedicado mantém somente o caminho operacional necessário.
9. funções privilegiadas usam exatamente `SECURITY DEFINER SET search_path = ''`, objetos qualificados e ACL explícita; testes cobrem `has_function_privilege` para todos os papéis.
10. referência privada em assignment entre usuários é rejeitada; texto livre continua permitido sem expor UUID privado.

**RPCs iniciais**

- `search_food_catalog(query, limit, cursor, kind)`
- `create_my_private_food(payload)`
- `update_my_private_food(food_id, payload)`

## Estratégia de rollback

- Antes de frontend consumir catálogo: rollback é remover a migration apenas em ambiente local não compartilhado ou criar migration inversa antes de aplicar remotamente.
- Depois de aplicação compartilhada: nunca apagar migration; criar migration compensatória que revoga RPCs, desativa fontes/lotes e preserva tabelas referenciadas.
- Frontend mantém feature gate/fallback local, permitindo desativar leitura do catálogo sem perder diário.
- Lote importado é revertido logicamente por `batch_id`; sob lock, ponteiros correntes são recalculados para a revisão mais recente ainda promovida, canônicos exclusivos só são arquivados sem revisão posterior/referência e registros históricos/source records permanecem auditáveis.
- Snapshot nunca é revertido por mudança de catálogo.
